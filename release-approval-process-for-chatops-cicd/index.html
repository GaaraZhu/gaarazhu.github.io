Release approval process for ChatOps driven CI/CD methodologies

Previously we discussed how to improve our CI/CD process through the ChatOps tool Bamboo-on-Teams. One command of it is to release a set of services in sequential batches. But the challenge is how to control the releases to critical environments like UAT and Production. This post shows how we implemented a release approval process for ChatOps driven CI/CD methodologies.

 

Solution


Bamboo-on-Teams is a powerful tool integrates the messaging platform Microsoft Teams and the CI/CD platform Atlassian Bamboo. It provides a set of predefined commands to be used through chatting messages in Teams channel which will be translated to corresponding Bamboo operations in Bamboo-on-Teams to be executed.

Since it already interacts with Bamboo, we can ultilise Bamboo to manage the releases instead of introducing a new datastore. The idea is to have a Bamboo plan to approve or deny the releases to certain environments through the customized build run.

Sequence diagram

steps

  1. Release manager approves the release

    1.1. click on the “Run customized…” option in plan page

    1.2. override variables to approve/deny the release to an environment

  2. DevOps start the release in Teams through Bamboo-on-Teams: @Bamboo release -s A,B;C,D;E -e prod

  3. Release manager denies any other releases after the release(same with step 1, just set the approved variable to “false”)

 

Bamboo Configurations

A simple build plan needs to be created without any SCM repositories.

  • Variables
    Two varaibles need to be added for this plan

 

  • Task
    For a success execution, Bamboo requires a mininum of one task configured for the plan which I used a script task

 

  • Permissions
    The Build plan permission should only be granted to release manager user or group.

 

Conclusion


With customized build run support in Bamboo, we implemented this approval process which secures our release actions through Bamboo-on-Teams.
